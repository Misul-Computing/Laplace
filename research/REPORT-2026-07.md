# Laplace Research Report - July 2026

External landscape audit (vLLM, llama.cpp, MLX, Apple-native stack) plus
internal bottleneck audit. Goal: find where Laplace must improve and how,
given the Mac-only pivot and the "THE Apple Inference Engine" mandate.

## 0. The honest starting point

Laplace is ~3.5x behind llama.cpp on its headline model. README claims
Gemma 4 26B MoE Q4 on M5 Pro = 18 tok/s. llama.cpp does Gemma4-26B-A4B
Q4_K_M at 63.9 tok/s on M3 Max (Hiesch benchmark), which is a slower
chip. The 26B-A4B MoE has 4B active params; at Q4 that is ~2 GB/token
to stream. M5 Pro bandwidth is ~120-150 GB/s, so the memory-bandwidth
ceiling is ~60-75 tok/s. llama.cpp sits at the ceiling. Laplace sits at
~25-30% of it.

Everything below is in service of closing that gap and then winning on
perf/W, which is the one axis nobody else competes on.

## 1. Internal audit (where Laplace's time goes)

### 1.1 The Metal detour was a dead end, and here is why

The last ~25 commits built a Metal GPU backend that underperformed and
got walked back to "CPU for layers, GPU for LM head only" at 3.4 tok/s.
Root cause, verified in the code:

- `matmul_gemm_batch` and `matmul_gemm_batch_async` in
  `src/matmul.cpp` are CPU stubs. They call `matmul_rows` sequentially
  and never dispatch to Metal. The entire "batch API / async / SSD
  prefetch overlap" commit chain ran on the CPU fallback path. The
  async overlap that the commit messages claim was not actually
  happening.
- The Metal kernel in `src/metal.mm` is a single universal GEMV with
  per-element dequant and no simdgroup matrix-multiply-accumulate
  (mma). A naive GEMV with per-element dequant cannot beat a tuned
  multi-threaded NEON CPU path at batch=1, because the GPU's compute
  density (simdgroup mma, tiled shared memory) is unused and command
  buffer dispatch overhead dominates at small M.
- Metal is gated behind `LAPLACE_METAL=1` and only used for the LM
  head. All layer matmuls run on the CPU SIMD path.

So the Metal experiments failed for the right reason: wrong kernel,
wrong dispatch, and the batch API was a stub. The GPU is not the
problem; the GPU integration was.

### 1.2 The real decode hot path

All layer matmuls (Q/K/V/O, FFN, MoE experts) run on the CPU SIMD path
in `src/matmul_simd.cpp`. The ARM half (line 600+) is solid: DOTPROD
`vdotq_s32` for int8, FP16 vector arithmetic, FMA. The x86 half (lines
10-599, ~590 lines) is dead code after the Mac pivot.

Attention (`src/arch_gemma4.cpp` lines 390-433) is a serial per-token
loop with a NEON-vectorized inner dot. O(seq_len) KV loads per query
head, no FlashAttention tiling. That is the long-context bottleneck.

Scheduling (`src/main.cpp` lines 461-496) is single-stream: one
prompt, one generation, batch=1 decode with small-M speculative
verify. No continuous batching, no multi-request scheduling.

### 1.3 What is genuinely strong (keep, do not touch)

- Power-aware thread scheduling: PowerMonitor, E-core/P-core QoS,
  futex parking via `os_sync_wait_on_address`. Nobody else has this.
  It is the 3-5x perf/W differentiator.
- TurboQuant KV compression and LaplaceKV paged cache. Novel, documented
  in the June report.
- FlashMoE SSD streaming for models larger than unified memory.
- WH-domain DeltaNet state. Novel, documented in the June report.

## 2. External landscape

### 2.1 vLLM - what to borrow, what to skip

vLLM is GPU-first (CUDA, NCCL, discrete GPU memory). The kernels are
not portable. The architecture is.

Borrow:
- Paged KV cache with block tables (Laplace already has LaplaceKV
  paged; the vLLM idea to confirm is block-table logical-to-physical
  mapping with reference counting for prefix sharing, 40-60% memory
  savings in chat workloads).
- Iteration-level scheduling with a token budget (continuous batching).
  Simplify to waiting/running/finished for batch <= 8; skip the
  swapped/preempted/skipped_waiting states.
- Automatic prefix caching (hash-based, full-block). 30-60% throughput
  gain on repeated system prompts.
- Chunked prefill (SARATHI, arXiv:2308.16369): split long prefills so
  decode latency stays bounded. Decode-prioritize.
- RECOMPUTE preemption only. SWAP is meaningless on unified memory.

Skip:
- CUDA Graphs, NCCL, Marlin/Machete kernels, FlashAttention CUDA
  kernels, SWAP preemption, distributed scheduling, model-based
  speculative decoding (EAGLE/draft) at batch=1.

### 2.2 llama.cpp - the competitor to beat, with numbers

Concrete numbers on Apple Silicon (from the research, cited there):
- Gemma4-26B-A4B Q4_K_M, M3 Max: 63.9 tok/s (with E2B draft: 206.5,
  3.2x).
- Qwen3.6-35B-A3B Q8, M5 Max: 105 tok/s with MTP, 93 without.
- Llama 3.1 70B Q4_K_M, M4 Max: 12.5 tok/s.
- MLX beats llama.cpp 15-20% on 8B, 35-45% on 70B. Gap collapses under
  14B where bandwidth dominates.

llama.cpp's Metal backend is well-tuned: separate GEMV (decode) and
GEMM (prefill) kernels, shared-mode MTLBuffer for zero-copy, Q4_K
repack with i8mm giving 1.62x on M4 Max. KV cache quantization:
q8_0-K + q4_0-V costs only 1.3% quality for half the V storage
(Discussion #23470).

llama.cpp's weaknesses (the gaps Laplace can take):
- No power-aware scheduling. Manual `--cpu-no-efficiency-cores` flag
  only. No runtime QoS, no thermal feedback, no battery awareness.
  MediaTek ARMv9 shows a 100x collapse when threads exceed big-core
  count; llama.cpp has no defense.
- Fused dequant+matmul is missing on Metal. Issue #23752: MTP
  speculative decoding is slower than baseline because the Q4_K
  pipeline is dequant (write f32) then matmul (read f32), two kernel
  launches. This is a known llama.cpp gap.
- GPU underutilization: Gemma 4 31B Q8 on M4 Max draws only 58W of
  145W system during prefill (Issue #22745). Compute headroom unused.
- ANE unused in mainline. Experimental PR #15262: ANE vision prefill
  7.2x faster than Metal on M2.

### 2.3 MLX - the Apple-native reference

MLX is the strongest Apple-native engine, C++ for the same reasons
Laplace is. Numbers: MLX 20-87% faster than llama.cpp under 14B;
speculative decoding 2.1x on M5 Max (Llama 3.1 70B + 1B draft).

What MLX does right (borrow):
- Unified memory: MTLResourceStorageModeShared, zero-copy, no host-
  device transfers. Laplace's mmap'd weights already do this on the
  CPU side; the Metal side needs to use shared-mode buffers, not the
  per-tensor copy cache the Metal detour built.
- Graph fusion for attention: `scaled_dot_product_attention` hits
  93-99% of matmul peak. This is the long-context prefill win.
- simdgroup_matrix for GEMM. On M5+, MPP TensorOps (mpp::tensor_ops)
  beats naive threadgroup-tiled GEMM by 2.9-5.5x (arXiv:2606.12765).
  simdgroup_matrix stays within ~20% of MPP even on M5.
- Dequant-fused matmul kernels, template-based, 2/3/4/5/6/8-bit.
- Affine quant format (Laplace already has an MLX affine type; confirm
  compatibility).

What MLX does not do (Laplace's wedge):
- No power-aware scheduling. No P/E core awareness, no thermal
  feedback, no battery mode. This is the 3-5x perf/W opportunity and
  nobody has it.
- MLX is a general array framework, not a dedicated inference engine.
  Lazy evaluation and graph build have overhead Laplace can skip with
  eager execution and specialized kernels.
- No advanced KV compression. MLX has basic quantization; Laplace has
  TurboQuant + LaplaceKV paged with 4-bit WARM / 2-bit COLD.

### 2.4 ANE reality check

ANE is not practical for 7B+ LLM inference with public APIs. CoreML
graph limit is ~1GB compiled weights (ANEF error -14 above that);
practical shard ceiling ~250MB. 8B needs multiple shards, each adding
2.3ms XPC round trip, capping throughput around 108 tok/s theoretical
before any real work. ANEMLL tops out at 8B and gets 47-62 tok/s on
Llama 3.2 1B vs ~204 tok/s on GPU.

ANE wins on perf/W (2W vs 20W, ~10x) for small models (1-3B) in
battery-critical always-on use. For interactive 7B+ inference, GPU
wins. Recommendation: do not pursue ANE for the headline path. Keep it
as a future option for a "battery sipper" mode on small models via
private APIs (Orion, arXiv:2603.06728), but it is not 1.0 work.

### 2.5 Metal SOTA

- simdgroup_matrix (8x8 tiles) is the workhorse on M1-M4. MPP
  TensorOps on M5+ is 1.05-1.21x faster than simdgroup_matrix, not a
  separate hardware unit.
- BF16 is emulated in software on M1/M2 (same speed as FP32). Use
  FP16 on M1-M4. BF16 native only on M5+.
- Metal 4 (M4+): unified compute encoder (compute + blits in one
  encoder), command barriers for fine-grained sync, decoupled command
  buffers for parallel encoding. Reduces encoder overhead.
- MTLResidencySet prevents GPU memory "unwiring" during idle (fixes
  the idle-throttling in llama.cpp PR #10119). Worth using.
- Shared-mode MTLBuffer for zero-copy. The Metal detour's per-tensor
  copy cache was the wrong model for unified memory.

## 3. Ranked improvement plan

Ordered by signal-to-effort, with the honest framing that closing the
bandwidth gap on the headline model is job one.

### 3.1 Fix CPU bandwidth utilization on the headline MoE path (job one)

The 18 vs 64 tok/s gap is a CPU-side bandwidth utilization problem,
not a missing-feature problem. The MoE decode streams 4B active params
per token; the CPU path is not saturating memory bandwidth. Likely
causes to profile and fix:
- Per-row dispatch in the scalar/GEMV path is not vectorized enough or
  is thrashing the thread pool at the wrong granularity.
- No fused dequant+dot for the K-quant formats in the ARM path (the
  x86 path has dot32_i8 with VNNI; the ARM path has dot16_i8 with
  DOTPROD but the K-quant dequant fusion needs checking).
- Thread pool overhead per matmul call at batch=1.

Verify target: Gemma4-26B-A4B Q4 on M5 Pro should hit 50+ tok/s CPU-
only before any GPU work. llama.cpp's 63.9 is the bar.

Effort: 1-2 weeks of profiling and kernel work. This is the highest-
signal work in the entire plan.

### 3.2 Do Metal right this time, or delete it

The Metal detour cost ~25 commits and landed at 3.4 tok/s. Two honest
options:

Option A (recommended): rebuild the Metal path with simdgroup_matrix
GEMM (not GEMV), shared-mode zero-copy MTLBuffer for weights (no copy
cache), and fused dequant+matmul kernels. Target the compute-bound
prefill and the LM head first, where the GPU wins. Leave decode on CPU
until the GPU kernel beats the tuned CPU path. This is what MLX does.

Option B: delete `src/metal.mm` and the Metal hooks entirely, commit
to CPU-only, and win on perf/W with the power-aware scheduler. Simpler,
fewer moving parts, and the CPU path is where the gap is anyway.

Pick one. Do not leave the current half-wired state where Metal is
gated off and the batch API is a CPU stub. That is the worst of both.

Effort: Option A is 3-4 weeks. Option B is 1 day.

### 3.3 Power-aware scheduling as the differentiator (the wedge)

This is the one thing neither llama.cpp nor MLX does, and it is the
3-5x perf/W path. Laplace already has the foundation (PowerMonitor,
E-core QoS, futex parking). The missing pieces:
- Phase-specific allocation: prefill (compute-bound) on all P-cores,
  decode (memory-bound) on E-cores alone or E + 1-2 P, attention on
  E-cores, sampling on a single core.
- Proactive thermal backoff before the OS intervenes (llama.cpp's M3
  Max throttling issue #10444 shows the OS intervention is rough).
- Battery vs AC continuous allocation, not just thread count.
- MTLResidencySet to prevent GPU memory unwiring during idle gaps.

Effort: 2-3 weeks. This is the publishable differentiator and the
"Apple Inference Engine" identity.

### 3.4 FlashAttention-style tiled attention on the long-context path

The current attention is a serial per-token loop with a NEON inner
dot. At 32K+ context this is the bottleneck. A tiled IO-aware
attention (FlashAttention algorithm, not the CUDA kernel) reduces KV
memory traffic and fuses softmax. On Metal this is MLX's
scaled_dot_product_attention which hits 93-99% of matmul peak. On CPU
it is a tiled GEMM-style attention.

Effort: 2 weeks. High value for long-context, medium for short.

### 3.5 Continuous batching + prefix caching (if server use is in scope)

Only if Laplace targets multi-request serving, not just batch=1 local.
vLLM's iteration-level scheduler with a token budget, simplified to
waiting/running/finished, plus hash-based prefix caching for repeated
system prompts. 30-60% throughput gain on chat workloads.

If the mandate is "THE Apple Inference Engine" for interactive single-
user use, skip this for 1.0. If it is a server too, this is a 3-4
week project.

Effort: 3-4 weeks. Defer unless serving is a goal.

### 3.6 Fused dequant+matmul for K-quant on Metal (if keeping Metal)

llama.cpp's Issue #23752 shows the two-step dequant+matmul pipeline is
a known gap on Metal. A single-pass fused kernel eliminates the
intermediate f32 buffer and one kernel launch. Estimated ~2x per-
matmul on Q4_K. This is the same fix Laplace's Metal path needs.

Effort: 1 week, only if keeping Metal.

## 4. What not to do

- ANE offload for 7B+ models. Not practical with public APIs. Defer
  to a future "battery sipper" mode for 1-3B models.
- EAGLE/draft-model speculative decoding at batch=1. The memory cost
  of a second model is not worth it for single-user interactive use.
  N-gram/prompt-lookup (already shipped) and Lookahead (already
  shipped) are the right CPU-feasible paths.
- Cross-platform support. The Mac-only pivot means deleting the x86
  SIMD, CPUID, Zen 4 tuning, and Linux/Windows roadmap. That is a
  real simplification. Do it.
- SWAP preemption, NCCL, CUDA Graphs, Marlin/Machete. All GPU-
  specific or server-specific, none apply.

## 7. LTO regression fix (commit 82a5f0b, 2026-07-03)

### 7.1 Scope and timeline

Six commits landed between Jul 2 and Jul 3, 2026:

    c04fbdc  Wire fused MoE kernels, add i8mm usdot, N_DST=2 dispatch
    d4f6f33  Optimize Gemma4 attention: contiguous FP16 streaming + head parallelism
    422b6db  Adaptive residency planning, streaming-aware MoE dispatch
    3f3dada  FlashMoE: multi-threaded pread page-in, adaptive mlock expert cache
    ef8df30  FlashMoE: increase expert cache retention to 16 tokens
    82a5f0b  Fix Q6_K offset folding regression, disable streaming MoE dispatch (the fix)

The first five introduced two independent bugs that both produced
garbage output from the 26B MoE model. The fix commit addresses both.

### 7.2 Bug 1: Q6_K offset folding (introduced by c04fbdc)

#### What the commit changed

c04fbdc rewrote the Q4_0, Q4_K, and Q6_K dot-product kernels in
`src/matmul_simd.cpp` to use i8mm `vusdotq_s32` (unsigned bytes dot
signed int8) instead of the previous approach of subtracting the
zero-point from weights before a signed `vdotq_s32` dot product.

The motivation was to eliminate `vsubq_s8` from the hot loop. Instead
of computing `dot(nibble - 8, xv)` with signed arithmetic, the new
code computed `dot(nibble, xv)` with mixed-sign usdot and subtracted
the zero-point correction as a scalar:

    sum((u - z) * xv) = sum(u * xv) - z * sum(xv)

For Q4_0 (z=8) this used the block sum `a.sum`. For Q6_K (z=32) this
used per-half sums `a.sum_lo` and `a.sum_hi` because Q6_K blocks
carry two scales per super-block (sc[2*g] and sc[2*g+1] for the two
halves), so the offset correction must be computed per-half:

    acc_offset += 32 * d * a.d * (sc[2*g] * a.sum_lo + sc[2*g+1] * a.sum_hi)

The ActQ8 struct gained two int16 fields:

    struct ActQ8 {
        float   d;
        float   sum;
        int16_t sum_lo;   // sum of qs[0..15]
        int16_t sum_hi;   // sum of qs[16..31]
        int8_t  qs[32];
    };

#### Why it broke

The per-half offset folding is numerically equivalent to direct
subtraction in exact arithmetic. It broke under optimization for
reasons that were not fully isolated. The folding path was tested in
isolation (computing sum_lo/sum_hi from saturated int8 values, from
pre-saturation int32 values, using dot16_i8 instead of usdot) and
none of these variations produced correct output. Only reverting to
the original direct -32 subtraction via `dot16_i8` with a `b32`
vector resolved the issue.

The most likely explanation is that the offset folding path has a
subtle interaction with the Q6_K block layout. Q6_K super-blocks are
256 elements, split into 4 groups of 64, each group split into 2
halves of 32. The scales sc[0..7] are packed in a packed-scale block.
The offset folding requires matching each half's activation sum with
the correct scale. An index mismatch in the scale/half pairing would
produce wrong results that look correct in simple test cases but fail
on real model weights.

A secondary issue was found in `quantize_act_q8_neon`: the `sum_lo`
and `sum_hi` fields were computed from post-saturation int8 values
(after `vqmovn_s32` narrowing) instead of pre-saturation int32
values. When activation values exceed [-127, 127], saturation clips
them, and the sum computed from clipped values is wrong. The Q4_0
offset correction uses `a.sum = d * (sum_lo + sum_hi)`, so a wrong
sum gives a wrong correction. This was fixed by computing the sums
from the int32 accumulators before narrowing:

    a.sum_lo = static_cast<int16_t>(vaddvq_s32(s0));
    a.sum_hi = static_cast<int16_t>(vaddvq_s32(s1));
    a.sum = d * static_cast<float>(a.sum_lo + a.sum_hi);

#### The fix

Reverted Q6_K to direct -32 subtraction. The weight values are
reinterpreted as signed and subtracted with `vsubq_s8(..., b32)`
where `b32 = vdupq_n_s8(32)`, then dotted with `dot16_i8` (signed
`vdotq_s32`). No offset correction is needed because the subtraction
happens before the dot product:

    int8x16_t w0 = vsubq_s8(vreinterpretq_s8_u8(u0), b32);
    int8x16_t w1 = vsubq_s8(vreinterpretq_s8_u8(u1), b32);
    int32x4_t p0 = dot16_i8(w0, xv0);
    int32x4_t p1 = dot16_i8(w1, xv1);
    // no acc_offset needed
    return hsum_f32x4(vaddq_f32(acc0, acc1));

Q4_0 was also reverted to the same pattern (subtract 8 before dot).
The `dot16_u8i8` helper and the i8mm `vusdotq_s32` path were removed
since they are no longer used.

The ActQ8 struct keeps the `sum_lo`/`sum_hi` fields (computed from
pre-saturation int32) because `a.sum` is still used by the Q4_0
offset correction path. The Q4_0 kernel still uses offset folding
(unchanged from before c04fbdc), and its correction uses `a.sum`
which is now computed correctly from pre-saturation values.

### 7.3 Bug 2: streaming MoE dispatch (introduced by 422b6db)

#### What the commit changed

422b6db added a streaming-aware MoE dispatch path in
`arch_gemma4.cpp`. When `FlashMoE::streaming_enabled()` returns true
(model does not fit in RAM, experts are on SSD), the MoE FFN switches
from the fused kernel to sequential per-expert matmuls via
`matmul_gemm_batch`. The intent was to give the OS a sequential
readahead pattern from SSD: each expert's weight slice is accessed
one at a time, instead of the fused kernel's parallel access across
all experts which causes 8-way random SSD reads.

The streaming path uses `expert_view()` to create a 2D Tensor view
into the 3D stacked expert tensor, then passes it to
`matmul_gemm_batch`:

    Tensor expert_view(const Tensor* stacked, int expert_idx) {
        Tensor v = *stacked;
        v.n_dims = 2;
        v.dims[0] = stacked->dims[0];  // N
        v.dims[1] = stacked->dims[1];  // K
        v.dims[2] = 0;
        v.dims[3] = 0;
        size_t per_expert = v.nbytes();
        v.data = stacked->data + expert_idx * per_expert;
        return v;
    }

    MatmulBatchSpec gu_specs[16];
    for (int k = 0; k < top_k; k++) {
        Tensor view = expert_view(W.moe_gate_up_exps, top_idx[k]);
        gu_specs[k] = {moe_in, &view, gu_bufs.data() + k * 2 * exp_inter,
                       H, 2 * exp_inter};
    }
    matmul_gemm_batch(gu_specs, top_k);

#### Why it broke

`matmul_gemm_batch` is a CPU stub that calls `matmul_rows` sequentially:

    bool matmul_gemm_batch(const MatmulBatchSpec* specs, int n) {
        for (int i = 0; i < n; i++)
            matmul_rows(specs[i].x, *specs[i].w, specs[i].y, 1,
                        specs[i].K, specs[i].N);
        return true;
    }

The fused path (`fused_moe_gemm_idx`) uses a different code path. It
calls `kernels::get_simd_moe_gemv()` which dispatches to
`moe_gemv_int<dot_row_q6_k_int_neon>`. This function quantizes the
activation once and does one `parallel_for` across all expert*column
pairs, indexing into the stacked tensor with `expert_idx[k] *
per_expert`.

The difference between the two paths:
- Fused: `moe_gemv_int` computes `per_expert = N * rb` where
  `rb = K / elements_per_block(type) * bytes_per_block(type)`. This
  is the byte size of one expert's weight row. It indexes
  `w + expert_idx[k] * per_expert + j * rb`.
- Batch: `expert_view` computes `per_expert = v.nbytes()` which is
  `dims[0] * dims[1] * element_size`. Then `matmul_rows` calls
  `matmul_rows(x, view, y, 1, K, N)` which dispatches to
  `gemm_simd` -> `gemm_int<dot_row_q6_k_int_neon>`.

Both paths should produce the same result if the tensor view is
correct. The bug is likely in `expert_view`: it sets `v.n_dims = 2`
and `v.dims[0] = stacked->dims[0]`, `v.dims[1] = stacked->dims[1]`.
But the stacked expert tensor is 3D with dims [N, K, n_experts] or
[n_experts, N, K] depending on the GGUF layout. If the dimension
ordering is wrong, `v.nbytes()` computes the wrong per-expert offset,
and the view points to the wrong memory.

The fused path avoids this issue entirely by computing the offset
from the weight type's block layout (`rb = K / elements_per_block *
bytes_per_block`), not from tensor dimensions. This is robust to
dimension ordering because it only depends on K and N.

This was not fully debugged. The fix disables the streaming path.

#### The fix

Both streaming MoE dispatch sites in `arch_gemma4.cpp` are gated with
`if (false && FlashMoE::streaming_enabled())`:

- Line 657: gate_up batch (matmul_gemm_batch with expert_view slices)
- Line 694: down batch (matmul_gemm_batch with expert_view slices)

The fused path is always taken instead:
- `fused_moe_gemm_idx` for gate_up (shared activation across experts)
- `fused_moe_gemm_multi` for down (per-expert activations)

#### Impact of the fix

The SSD streaming optimization is lost. The 26B model (16.8 GB)
exceeds the 22 GB RAM budget after OS overhead, so
`FlashMoE::streaming_enabled()` returns true. With the streaming path
disabled, the fused kernel is used instead. The fused kernel accesses
all 8 selected experts' weights in parallel via `parallel_for`, which
causes 8-way random SSD access when experts are not in RAM.

In practice, the model still works because the OS page cache and
FlashMoE's `MADV_WILLNEED` prefetch (pagein_all_mt) pull the active
experts into RAM before the matmul. The prefetch is still active
(lines 640-654 in arch_gemma4.cpp). What is lost is the sequential
access pattern that would let the OS do efficient readahead during
the matmul itself, overlapping I/O with compute.

The net effect on this machine (26B model, 22 GB RAM): the model
works correctly but may be slower than the streaming path would be
for cold experts. The prefetch path mitigates this for warm workloads.

### 7.4 What was NOT broken

The attention rewrite from d4f6f33 (contiguous FP16 streaming, head
parallelism, GQA amortization) was initially suspected but works
correctly. It was tested in isolation and produces correct output.
The fast path (M=1, FP16/FP32 KV cache) is taken as confirmed by
debug output. The `head_k16`, `head_v16`, `dot_f16`, and `axpy_f16`
methods in kvcache.cpp and ops.h are present and functional.

The `N_DST=2` blocking for `gemm_int` and `gemm_f` in
`matmul_simd.cpp` (also from c04fbdc) was tested and works correctly.

The `model.cpp` adaptive residency planning (from 422b6db) works
correctly. The 26B model loads, pins dense weights, and streams
experts as intended.

### 7.6 What is still broken

#### 1. Streaming MoE dispatch (matmul_gemm_batch)

The sequential per-expert path produces incorrect output. The fused
path works. Both are gated with `if (false && ...)` in
`arch_gemma4.cpp` lines 657 and 694.

To fix: debug why `matmul_gemm_batch` with `expert_view` slices gives
different results than `fused_moe_gemm_idx` with the full stacked
tensor. The most likely cause is `expert_view` computing the wrong
per-expert byte offset from tensor dimensions, while the fused path
computes it from the weight type's block layout. Check the dimension
ordering of the stacked expert tensor ([n_experts, N, K] vs [N, K,
n_experts]) and verify that `v.nbytes()` in `expert_view` gives the
same value as `N * rb` in `moe_gemv_int`.

Key architectural differences from the 26B:

- 48 layers vs 30. hidden=3840 vs 2816. inter=15360 vs 2112.
- Dense FFN (no MoE) vs 128 experts (8 active).
- Global layers have 1 KV head (head_count_kv=1) vs 26B's 2.
  This gives gqa=16 for global layers (16 query heads / 1 KV head)
  vs gqa=8 for the 26B.
- Same SWA pattern: [1,1,1,1,1,0] repeating. Same head_dim: 512
  global, 256 sliding. Same rope freqs.

The arch_gemma4.cpp code parses these features (lines 134-158: per-
layer KV head counts, SWA pattern, head_dim per layer type). The
attention_batch function (line 306) uses per-layer Hq, Hk, Dh, gqa,
and swa_window. The KV cache is allocated with max dims (n_kv_heads=8,
head_dim=512). Sliding layers pad to Dh_store=512 when storing.

The code looks correct on inspection. The bug is likely in one of:
- Q/K/V projection dimensions for sliding vs global layers. The Q
  weight tensor for sliding layers is [16*256, 3840] = [4096, 3840]
  and for global layers is [16*512, 3840] = [8192, 3840]. The code
  computes qn = Hq * Dh per layer, which should match. But if the
  weight tensor's dims[0] is used instead of qn, there could be a
  mismatch.
- The QKV matmul uses matmul_gemm_batch (line 344), which calls
  matmul_rows with K=hidden, N=qn. If the weight tensor's actual N
  (dims[0]) differs from qn, the matmul reads the wrong number of
  rows. This would be a bug if the GGUF stores Q weights with
  dims[0] = max(qn_global, qn_sliding) = 8192 for all layers, but
  the code passes N=4096 for sliding layers.
- RoPE application: sliding layers use rope_pairs_swa_ (128 pairs
  for head_dim=256) and global layers use rope_pairs_full_ (256
  pairs for head_dim=512). The rope tables are precomputed with the
  correct freq bases (10000 for SWA, 1e6 for global). This looks
  correct.
- The dense FFN path (lines 764-802) uses matmul_gemm_batch for
  gate/up/down projections. This is the same matmul_gemm_batch that
  works for the 26B's QKV projection, so it should be fine. But the
  12B's intermediate=15360 is much larger than the 26B's 2112, which
  could expose a buffer size issue if ffn_buf is sized incorrectly.
  The reserve function (line 245) sizes ffn_buf as
  max(intermediate, expert_inter*2). For the 12B (no MoE),
  expert_inter=0, so ffn_buf = max(15360, 0) = 15360. This looks
  correct.

This bug needs systematic debugging: run with LAPLACE_NOSIMD=1 to
isolate SIMD vs scalar, check tensor dimensions at load time, and
compare intermediate activations against a reference implementation
(llama.cpp or MLX) at each layer.
